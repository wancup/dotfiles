# Fallowレビュー

`fallow review`を使い、変更差分の構造的リスクを日本語で整理する。

## 1. 引数の解釈

`$ARGUMENTS`を空白区切りの位置引数として解釈する。

- 第1引数がある場合: 分岐元refとして扱う
- 第1引数がない場合: 分岐元は指定せず、`fallow review`の自動検出に任せる

第2引数以降は使用しない。

## 2. レビュー結果の取得

分岐元refが指定されている場合は、以下を実行する。

```bash
fallow review --base <分岐元ref> --format json --quiet 2>/dev/null || true
```

分岐元refが指定されていない場合は、以下を実行する。

```bash
fallow review --format json --quiet 2>/dev/null || true
```

`fallow review`はレビュー用の助言を返すコマンドであり、終了コードだけで失敗扱いしない。JSON出力の内容を確認して判断する。

## 3. レビュー観点

fallowのJSON結果をもとに、以下を中心に確認する。

- 変更ファイルのうち、fallowが注目すべきと判断している箇所
- dead code、複雑度、重複など、決定的に検出できる懸念
- 影響範囲や依存関係の広がりが大きい箇所
- public API、依存関係、境界越えなど、構造上の判断が必要な箇所
- fallowが検出した内容がない場合でも、解析対象や前提に問題がないか

ただし、実際の差分本文を読んだ意味的レビューは行わない。fallowの結果から言える範囲を超えて断定しない。

## 4. レビュー結果の出力

以下のフォーマットで日本語でまとめる。

### サマリー

fallow reviewの結果を1〜2文で要約する。

### 注目箇所

fallowが優先的に見るべきとしているファイルや領域があれば、理由とともに箇条書きで示す。該当がなければ「特になし」とする。

### 構造的な懸念

dead code、複雑度、重複、依存関係、public API、境界越えなどの観点で懸念があれば、該当ファイルやsignal idなど、fallow出力に含まれる根拠とともに記載する。該当がなければ「特になし」とする。

### 総合判定

以下のいずれかで判定する。

- LGTM: fallowの結果から重大な構造的懸念が見つからない
- 要修正: dead code、複雑度、重複など、修正すべき決定的な問題がある
- 要議論: public API、依存関係、境界越えなど、設計判断が必要な問題がある

判定理由を1〜2文で添える。
