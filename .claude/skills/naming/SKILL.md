# 命名スキル

## 入力

`$ARGUMENTS`

## 手順

1. **入力の解釈**
   - ファイルパスが含まれる場合: Readツールで該当コードを読み取り、コンテキストを把握する
   - それ以外: 渡された説明文から命名対象の意味・用途・文脈を理解する

2. **命名対象の分類**
   - 変数名、関数名、クラス名、定数名、型名など、対象の種類を判断する
   - 指定がなければ文脈から推測し、最も適切な種類で提案する

3. **命名候補の出力**
   - **5つ以上** の命名候補を提案する
   - 各候補について以下を記載する:
     - 候補名（camelCase / snake_case / PascalCase は対象の種類と言語慣習に合わせる）
     - ニュアンスの説明（その名前が読み手にどういう印象・意味を伝えるか）
     - 適する場面（どういうコンテキストで最適か）

## 出力フォーマット

```
## 命名候補: <対象の簡潔な説明>

| # | 候補名 | ニュアンス | 適する場面 |
|---|--------|-----------|-----------|
| 1 | `名前` | 説明 | 場面 |
| 2 | `名前` | 説明 | 場面 |
| ... | ... | ... | ... |

### 推奨
最も汎用的でおすすめの候補とその理由を1〜2文で述べる。
```

## ルール

- 言語やフレームワークが特定できる場合、その慣習に従った命名規則（camelCase, snake_case等）で提案する
- 短すぎる略語（`tmp`, `val`等）は避け、意図が明確に伝わる名前を優先する
- 英語の命名を基本とするが、各候補のニュアンス説明は日本語で行う
- 類義語の微妙な違い（例: get vs fetch vs retrieve）を明確に説明する
- 既存コードのファイルパスが提供された場合、周囲のコードの命名規則に合わせる
